### 1️⃣ Learned What a Network ACL (NACL) Is

Studied AWS Network ACL fundamentals.

Learned:
- NACL = Network Access Control List
- NACL acts as a subnet-level firewall
- NACL controls inbound and outbound traffic for an entire subnet

Understood:
- NACLs protect all resources inside a subnet
- NACLs operate separately from Security Groups

Visualized:

```text
Internet
    ↓
NACL
    ↓
Subnet
    ↓
EC2 Instance
```

Learned:
- NACLs are an additional network security layer in AWS

---

### 2️⃣ Learned Difference Between Stateful and Stateless Firewalls

Studied firewall behavior differences.

Learned:
- Security Groups are stateful
- NACLs are stateless

Understood:

Security Group behavior:

```text
Inbound Allowed
       ↓
Return Traffic Automatically Allowed
```

NACL behavior:

```text
Inbound Allowed
       ↓
Outbound Must ALSO Be Allowed
```

Visualized:

```text
Request → Allowed
Response → Must Be Explicitly Allowed
```

Learned:
- Stateless firewalls require rules for both directions of communication

---

### 3️⃣ Compared NACLs vs Security Groups

Studied architectural differences.

| Feature | Security Group | NACL |
|---|---|---|
| Level | Instance | Subnet |
| Stateful | Yes | No |
| Allow Rules | Yes | Yes |
| Deny Rules | No | Yes |
| Rule Processing | All rules | Lowest rule number first |
| Applies To | ENI/EC2 | Entire Subnet |

Understood:
- Security Groups provide instance-level filtering
- NACLs provide subnet-wide traffic control

Visualized:

```text
NACL → Subnet Protection
SG → Instance Protection
```

Learned:
- Enterprise systems use both together

---

### 4️⃣ Learned NACL Rule Evaluation Order

Studied AWS rule processing behavior.

Learned:
- NACL rules are processed from lowest rule number first
- First matching rule wins

Example:

```text
90 → DENY
100 → ALLOW
```

Understood:
- Rule order is extremely important

Visualized:

```text
AWS Checks:
Rule 90
   ↓
Rule 100
   ↓
Rule 110
```

Learned:
- Incorrect rule ordering can unintentionally allow traffic

### 5️⃣ Learned About Ephemeral Ports

Studied TCP client return traffic behavior.

Learned:
- Clients use temporary random ports during communication
- Ephemeral ports range:

```text
1024-65535
```

Understood:
- Web responses often return through ephemeral ports
- Blocking ephemeral ports breaks connections

Visualized:

```text
Browser → Server:80
Server → Browser:53211
```

Learned:
- Ephemeral ports are critical for stateless firewall communication
